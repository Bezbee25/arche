# Terraform Best Practices

description: IaC standards for Terraform: state management, IAM least-privilege, security scanning, testing, naming
tags: devops, terraform, iac, infrastructure, aws, azure, gcp, security, tfsec

Infrastructure as Code standards for maintainable, secure, and testable Terraform.

## Project Structure

```
infra/
├── modules/
│   ├── networking/
│   ├── compute/
│   └── database/
├── environments/
│   ├── dev/
│   ├── staging/
│   └── prod/
└── shared/
```

- Separate state per environment and component
- Keep modules focused on a single infrastructure concern
- Avoid single-resource modules — group related resources

## State Management

- Use remote backends (S3 + DynamoDB, Azure Blob + Table, GCS) — never commit state files
- Enable state locking to prevent concurrent modifications
- Enable state encryption at rest
- Use workspaces sparingly — prefer separate state files per environment

## Variables & Outputs

- Never hardcode values that vary between environments
- Use `sensitive = true` for secrets and mark outputs as sensitive:
  ```hcl
  variable "db_password" {
    type      = string
    sensitive = true
  }
  ```
- Document all variables with `description`
- Use type constraints: `type = string`, `type = list(string)`, `type = object({...})`

## Security

- Run security scanners in CI: `tfsec`, `checkov`, `trivy config .`
- Apply IAM least-privilege: no `*` actions, specify exact resources
- Enable CloudTrail/audit logging for all provider accounts
- Use KMS encryption for storage resources (S3, RDS, EBS)

## Code Quality

- Run `terraform fmt` before committing — enforce in CI
- Sort arguments alphabetically within blocks
- Use `terraform validate` and `tflint` in CI
- Add `lifecycle` blocks for sensitive resources:
  ```hcl
  lifecycle {
    prevent_destroy = true
    ignore_changes  = [tags["last-modified"]]
  }
  ```

## Modules

- Use semantic versioning for module references: `version = "~> 3.0"`
- Pin to specific versions in production environments
- Document modules with `README.md` generated by `terraform-docs`
- Avoid module dependencies that create circular references

## Testing

- Write unit tests using `.tftest.hcl` files:
  ```hcl
  run "validate_input" {
    command = plan
    assert {
      condition     = var.instance_type != ""
      error_message = "instance_type must not be empty"
    }
  }
  ```
- Test both positive (valid config) and negative (invalid config) scenarios
- Use `terratest` (Go) for integration tests against real infrastructure

## Naming Conventions

- Use `snake_case` for all resource names, variables, and outputs
- Include environment and region in resource names where applicable: `prod-us-east1-db`
- Tag all resources consistently:
  ```hcl
  tags = {
    Environment = var.environment
    Project     = var.project_name
    ManagedBy   = "terraform"
    Owner       = var.team
  }
  ```

## CI/CD Integration

- Plan on PRs, apply on merge to main
- Store plans as artifacts and apply the exact same plan
- Require manual approval for production applies
- Use OIDC for provider authentication — no long-lived credentials
